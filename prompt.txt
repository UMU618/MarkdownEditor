设计并开发一个简洁的 Markdown 编辑器应用。该编辑器应支持标准 Markdown 语法的实时预览功能，包括但不限于标题、列表、粗体、斜体、链接、图片、代码块、引用等元素的正确渲染。实现基本编辑功能，如文本输入、删除、复制粘贴，并确保编辑区域与预览区域的内容同步更新。提供语法高亮显示，增强代码块的可读性。设计简洁直观的用户界面，包含编辑区和预览区的合理布局（可考虑分屏或切换模式）。确保编辑器在主流浏览器（Chrome、Firefox、Safari、Edge）中具有良好的兼容性和响应式表现，支持不同屏幕尺寸的适配。实现基本的文件操作功能，如新建、保存 Markdown 文件，以及导出 HTML 文件功能。添加必要的用户体验优化，如状态栏显示编辑状态等。

代码实现使用 C++ 20，遵守基于 Google C++ Cooding Style 的代码规范，下面列举几个编码规范的例子，没有提到的，则严格按照 Google C++ Cooding Style：

1. 所有条件语句块必须使用花括号{}进行显式包裹，无论该语句块包含单行代码还是多行代码。这一规范适用于所有条件分支结构（if、for、while），确保代码的可读性、可维护性和避免潜在的逻辑错误。
2. 成员变量使用 snake_case_，比如常见的 MFC 风格变量 m_view，应该改为 view_。模板产生的代码里可能有这种风格的命名，见到就重构，注意重命名过程中需保证代码功能不受影响，所有引用该成员变量的位置均需同步更新。
3. 比较两个变量时，优先使用 <，这样可以更清晰地看清谁是小的一个，比如说 if (a > b) 这样的写法应该改为 if (b < a)，它们是等价的，但是对人类来说后者更容易理解。
4. 应该使用 #pragma once，这和 Google C++ Cooding Style 的建议不同，但这个工程是 Windows 专用的，#pragma once 更省事。

Important implementation note:

1. 本工程采用 vcpkg 作为依赖库管理工具，用于统一管理、安装和配置项目所需的第三方库。在开发过程中，所有外部依赖项均应通过 vcpkg 进行获取和管理，以确保开发环境的一致性和依赖版本的可控性。vcpkg 的根目录已配置为工程的 src 目录，相关配置文件路径为 src\vcpkg.json。

2. 此应用程序的用户界面（UI）是使用 Windows 模板库（WTL）框架开发的，并且明确不是使用微软基础类库（MFC）实现的。所有与 UI 相关的开发、修改和维护活动都必须遵循 WTL 编程模式、惯例和最佳实践，而非 MFC 方法。在整个开发周期中，应清楚理解并遵循这一区别，以确保与现有代码库架构保持一致。尤其需要注意：WTL 是基于 ATL 的，需要分辨清楚两者的命名空间。例如，CWindow 类属于 ATL 命名空间（ATL::CWindow），而非 WTL 命名空间（WTL::CWindow）。开发过程中应确保所有类和组件的命名空间引用准确无误，避免因命名空间混淆导致的编译错误或运行时异常。在代码审查和调试阶段，需重点检查此类命名空间使用情况，确保遵循 WTL 与 ATL 的正确命名空间规范。

3. Markdown 转 HTML 的库，指定为 Hoedown，而不是更常见的 MD4C，它的代码位于 src\third_party\hoedown。该解决方案应包含与 Hoedown 库的正确集成，处理标准 Markdown 语法转换，并为无效的 Markdown 输入包含错误处理。确保实现遵循库使用的最佳实践，包括必要的 Hoedown 资源初始化和清理，并生成有效的、格式良好的 HTML 输出，准确地表示原始 Markdown 内容。

4. 在项目开发过程中，应优先使用已在 src\vcpkg.json 中配置的 boost、ms-gsl、wil 等指定库。以 UTF-8 和 UTF-16 编码转换功能为例，必须使用 boost-nowide 库提供的标准转换方法实现，确保编码转换的准确性和兼容性。在实现过程中，需遵循各库的官方文档规范，确保代码符合项目依赖管理策略，避免引入未在 vcpkg.json 中声明的第三方库。
